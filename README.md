# ch-migrate

A Rust library and CLI for running ClickHouse schema migrations, inspired by [sqlx::migrate!](https://docs.rs/sqlx/latest/sqlx/macro.migrate.html).

## Features

- **Compile-time embedding** via `migrate!()` proc macro — SQL files are baked into your binary
- **Runtime discovery** via CLI — apply migrations from a directory
- **SHA-256 checksums** — detects if a previously applied migration file has been modified
- **ON CLUSTER support** — optional `ReplicatedMergeTree` tracking table for clustered setups
- **Forward-only** — no down migrations; ClickHouse DDL isn't transactional anyway

## Quick start

### As a library

Add to your `Cargo.toml`:

```toml
[dependencies]
ch-migrate = "0.1"
clickhouse = "0.14"
```

Create a `migrations/` directory next to your `Cargo.toml` with `.sql` files:

```
migrations/
  0001_create_events.sql
  0002_add_user_id.sql
```

Embed and run them:

```rust
use ch_migrate::{migrate, Migrator};

static MIGRATOR: Migrator = migrate!();

#[tokio::main]
async fn main() {
    let client = clickhouse::Client::default()
        .with_url("http://localhost:8123");

    let report = MIGRATOR.run(&client).await.unwrap();
    println!("Applied {} migrations", report.applied);
}
```

### As a CLI

```bash
cargo install ch-migrate-cli

# Create a new migration
ch-migrate add create_events_table

# Apply pending migrations
ch-migrate run --url http://localhost:8123 --database mydb

# Show status
ch-migrate info --url http://localhost:8123 --database mydb
```

## Migration files

Files must follow the naming convention `{VERSION}_{DESCRIPTION}.sql`:

- **VERSION**: a positive integer (recommended: `YYYYMMDDHHMMSS` timestamp)
- **DESCRIPTION**: underscore-separated words

Examples:
```
20260401120000_create_events.sql
20260401120100_add_user_id_column.sql
```

### Idempotent DDL

Since ClickHouse has no DDL transactions, use idempotent statements so retries are safe:

```sql
CREATE TABLE IF NOT EXISTS events (
    event_id UInt64,
    timestamp DateTime
) ENGINE = MergeTree()
ORDER BY (timestamp, event_id);
```

### Multi-statement migrations

Multiple statements separated by `;` are executed individually. Note: the splitting is naive and does not handle semicolons inside string literals. Prefer one statement per file.

### Adding new migrations with `migrate!()`

The `migrate!()` proc macro discovers migration files at compile time. Cargo cannot natively track directory contents for changes, so adding a new `.sql` file will **not** automatically trigger a rebuild of the file that calls `migrate!()`. After creating a new migration, run `touch` on the file containing the macro invocation, or use `cargo clean`, to force a rebuild. Existing files are tracked via `include_str!`, so editing a previously-discovered migration does invalidate the cache normally.

## Configuration

### CLI flags / environment variables

| Flag | Env var | Default |
|------|---------|---------|
| `--url` | `CH_MIGRATE_URL` | `http://localhost:8123` |
| `--database` | `CH_MIGRATE_DATABASE` | — |
| `--user` | `CH_MIGRATE_USER` | — |
| `--password` | `CH_MIGRATE_PASSWORD` | — |
| `--cluster` | `CH_MIGRATE_CLUSTER` | — |
| `--table-name` | — | `_ch_migrations` |
| `-d, --migrations-dir` | — | `migrations` |

### ON CLUSTER support

Pass `--cluster <name>` (CLI) or call `.with_cluster("name")` (library) to enable:

- Tracking table is created with `ON CLUSTER` and `ReplicatedMergeTree`
- Your migration SQL should include `ON CLUSTER` clauses where needed

## Tracking table

Applied migrations are tracked in `_ch_migrations` (configurable):

```sql
CREATE TABLE IF NOT EXISTS _ch_migrations (
    version          Int64,
    description      String,
    checksum         String,
    installed_on     DateTime DEFAULT now(),
    success          Bool,
    execution_time_ms UInt64
) ENGINE = MergeTree()
ORDER BY version
```

## Concurrency

There is no distributed locking. Do not run migrations concurrently from multiple processes.

## Server settings assumed

The migration runner inserts tracking-table rows using single-quoted SQL string literals with backslash escaping. This assumes the ClickHouse server has the default `backslash_is_escape = 1` session setting. If you have changed this setting globally, the description column for migrations whose names contain backslashes may store incorrectly. Migration filenames generated by `ch-migrate add` cannot contain backslashes, so this only matters for programmatically constructed migrations.
